## Milestone 1: PLG GTM Design - 1. Discovery, Pricing & Journey Architecture
**Tag:** `plg-gtm`
**Description:** Strategic discovery, pricing model design, customer journey mapping, and system architecture for PLG motion
**Hours:** 60h

### Task List: (PLG GTM Design) 1. Discovery & Pricing Strategy
**Contains:** Parts 1-2

| Task | Est | Description |
|------|-----|-------------|
| 1. Conduct Stakeholder Discovery Sessions | 4h | Meet with key stakeholders across GTM functions to understand current state, pain points, and PLG objectives. End state: Documented stakeholder requirements and success criteria for PLG motion.<br /><br />• Schedule 1:1 discovery calls with VP Sales, VP Marketing, VP Product, VP CS, and Finance lead<br />• Document current GTM motion (fully sales-led vs. any existing self-serve elements)<br />• Capture specific pain points each function faces with current motion<br />• Identify existing tools and systems in play across all functions<br />• Align on PLG objectives: is this additive (new segment) or transformational (shift primary motion)?<br />• Document constraints (budget, timeline, engineering capacity) |
| 2. Analyze Market and TAM for PLG Fit | 3h | Validate that PLG is appropriate for target market segments and identify which customer profiles are best suited for self-serve. End state: PLG market map identifying target segments and expected behavior patterns.<br /><br />• Review ICP documentation and segment by company size/complexity<br />• Identify which segments have low-touch buying behavior (SMB vs. enterprise)<br />• Research competitor PLG motions in the space for benchmarking<br />• Map expected user personas who would enter via PLG (end users vs. buyers)<br />• Document segment-specific requirements (e.g., SMB self-serve, Mid-market hybrid, Enterprise sales-led) |
| 3. Finalize Growth Model PLG Targets | 3h | Establish specific KPIs and targets for the PLG motion aligned to overall company growth model. End state: Documented PLG targets integrated into growth model with clear metrics definitions.<br /><br />• Review existing growth model and EOY bookings targets<br />• Define PLG-specific metrics: sign-up volume, activation rate, free-to-paid conversion, PQL volume<br />• Set targets for each metric based on market sizing and benchmarks (3-5% free-to-paid is best-in-class)<br />• Align Product team on what "activation" means (specific feature usage thresholds)<br />• Define Net Revenue Retention (NRR) targets for PLG cohorts (aim for 120%+)<br />• Document how PLG metrics roll up to overall bookings targets |
| 4. Define Pricing Model Structure | 4h | Design the pricing architecture including free tier, paid tiers, and enterprise pricing relationship. End state: Documented pricing model with clear tier definitions and upgrade paths.<br /><br />• Evaluate freemium vs. free trial vs. hybrid approach based on product complexity<br />• Define free tier value proposition (enough to hook, not enough to fully satisfy)<br />• Design paid tier structure (1-3 self-serve tiers is optimal)<br />• Document feature gates between tiers (what triggers upgrade consideration)<br />• Define relationship between PLG pricing and enterprise agreement pricing<br />• Identify any cannibalization risks between PLG and existing sales-led deals |
| 5. Map Upgrade Triggers and Expansion Paths | 3.5h | Define specific usage patterns and behaviors that indicate upgrade readiness. End state: Documented PQL criteria and expansion trigger definitions ready for implementation.<br /><br />• Define Product Qualified Lead (PQL) criteria based on usage thresholds<br />• Map user behaviors that correlate with upgrade intent (seats added, features tried, usage frequency)<br />• Design expansion triggers for sales outreach (e.g., team size > X, feature limit hit)<br />• Define hand-off timing from self-serve to sales-assisted<br />• Document upsell paths (same user upgrading) vs. expansion paths (new users in same account)<br />• Align Sales on when to engage vs. when to let self-serve continue |
| 6. Finalize Pricing Implementation Requirements | 2.5h | Document technical and operational requirements for pricing implementation. End state: Requirements document ready for billing system implementation.<br /><br />• Document pricing display requirements (public pricing page, in-app pricing)<br />• Define billing frequency options (monthly, annual, usage-based components)<br />• Identify discount and promotion capabilities needed<br />• Document enterprise agreement exceptions and override requirements<br />• Define revenue recognition requirements with Finance<br />• Identify ERP integration requirements for revenue reporting |

### Task List: (PLG GTM Design) 2. Journey & Architecture
**Contains:** Parts 3-4

| Task | Est | Description |
|------|-----|-------------|
| 7. Map Pre-Acquisition Journey | 3.5h | Document all touchpoints and paths a user takes before signing up for the product. End state: Pre-acquisition journey map with channel attribution points.<br /><br />• Identify all acquisition channels (organic, paid, referral, product viral loops)<br />• Map website conversion paths to sign-up<br />• Document trial/freemium sign-up flow and required information<br />• Define attribution tracking requirements for each channel<br />• Map content touchpoints that support awareness and consideration<br />• Document competitive evaluation stage if applicable |
| 8. Map Onboarding and Activation Journey | 4h | Design the complete onboarding experience from sign-up to activation milestone. End state: Onboarding journey with activation gates and drop-off intervention points.<br /><br />• Define the critical "aha moment" that indicates activation<br />• Map steps from sign-up to activation (aim for minutes/hours, not days)<br />• Identify onboarding friction points and required improvements<br />• Design intervention triggers for users who stall (emails, in-app prompts)<br />• Document onboarding variations by user persona if needed<br />• Define success criteria for onboarding complete |
| 9. Map Conversion and Expansion Journey | 4h | Document the journey from free user to paying customer and subsequent expansion. End state: Conversion and expansion journey with clear handoff points to sales.<br /><br />• Map free-to-paid conversion triggers and prompts<br />• Design upgrade flow (in-app, checkout, sales-assisted options)<br />• Document expansion journey (adding seats, upgrading tiers)<br />• Define when sales involvement begins (PQL threshold, deal size, enterprise signals)<br />• Map hand-off process from product to sales<br />• Document post-conversion journey including renewal touchpoints |
| 10. Map Decision Trees and Outcomes | 3.5h | Document all possible user decisions and outcomes throughout the journey. End state: Comprehensive decision tree covering all user paths.<br /><br />• Map all decision points where users choose a path (upgrade, stay free, churn)<br />• Document outcomes for each decision (what happens next in system)<br />• Identify churned user re-engagement paths<br />• Define dormant user reactivation triggers<br />• Map failed payment and recovery flows<br />• Document account deletion/downgrade processes |
| 11. Audit Current System Landscape | 3.5h | Document all existing systems and their capabilities relevant to PLG motion. End state: Current state system map with identified gaps.<br /><br />• Inventory all current GTM systems (CRM, marketing automation, CS platform)<br />• Document current data flows between systems<br />• Identify gaps in current stack for PLG requirements (product analytics, usage tracking)<br />• Assess CRM readiness for PLG records (can it handle high-volume, low-touch leads?)<br />• Evaluate billing system capabilities for self-serve transactions<br />• Document integration limitations and technical debt |
| 12. Design Target System Architecture | 4h | Create the target state architecture for PLG motion with all required systems and integrations. End state: System architecture diagram with integration requirements.<br /><br />• Define product analytics/usage tracking solution (Amplitude, Mixpanel, Heap)<br />• Design usage data pipeline from product to CRM/data warehouse<br />• Map PQL scoring model and where it lives (CRM, separate tool)<br />• Define reverse ETL requirements for syncing enriched data to tools<br />• Design self-serve billing integration (Stripe, Chargebee, Recurly)<br />• Document required CRM customizations for PLG records and workflows |
| 13. Map Data Model and Object Relationships | 4h | Define how PLG data objects relate to existing CRM data model. End state: Data model documentation showing object relationships.<br /><br />• Define PLG-specific objects (Product Users, Usage Events, PQLs)<br />• Map relationship between Product Users and CRM Contacts/Leads<br />• Design account hierarchy for PLG (self-serve accounts vs. enterprise accounts)<br />• Define data matching and deduplication rules<br />• Document field mappings from product to CRM<br />• Design unified person/account model across systems |
| 14. Define Data Governance and Definitions | 3.5h | Establish standardized definitions for all PLG metrics and data elements. End state: Data dictionary with agreed definitions across all teams.<br /><br />• Define MQL, SQL, PQL criteria and document formally<br />• Create activation definition with specific thresholds<br />• Define conversion event and timestamp<br />• Document expansion and churn definitions<br />• Align all teams on metric calculations<br />• Create data dictionary and distribute to teams |
