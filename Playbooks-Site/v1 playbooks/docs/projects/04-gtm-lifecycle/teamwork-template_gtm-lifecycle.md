# GTM Lifecycle - Project Details / Task List

**Tag:** `gtm-lifecycle`
**Total Hours:** 38h

## Milestone: GTM Lifecycle
**Description:** A foundational RevOps project that designs and implements the complete journey stages a lead or account travels from initial awareness through becoming a customer—with clear entry criteria, automated transitions, and date/time stamps configured in the CRM.

---

### Task List: (GTM Lifecycle) 1. Assessment & Design
**Contains:** Parts 1-2

| Task | Est | Description |
|------|-----|-------------|
| 1. Conduct Stakeholder Interviews | 2h | Meet with marketing, sales, and customer success leaders to understand their current lifecycle definitions and handoff processes. End state: Interview notes capturing each team's perspective on stage definitions and pain points.<br /><br />• Schedule 30-45 minute interviews with VP Marketing, VP Sales, and CS leader<br />• Document each stakeholder's current understanding of lifecycle stages (MQL, SQL, SAL, etc.)<br />• Capture their definition of "qualified"—what makes a lead ready for handoff<br />• Identify where they see leads falling through the cracks or getting stuck<br />• Note any existing SLAs or handoff agreements (formal or informal)<br />• Document friction points between teams regarding lead quality or timing |
| 2. Audit Existing CRM Configuration | 2.5h | Review current CRM fields, picklist values, and automation rules related to lead/contact/account stages. End state: Technical inventory of all lifecycle-related fields and automation in the CRM.<br /><br />• Pull list of all Lead Status and Contact Status picklist values from CRM<br />• Identify any Account-level lifecycle fields (e.g., Account Stage, Customer Status)<br />• Document existing automation rules that change lifecycle stages<br />• Review any validation rules that govern stage progression<br />• Check for existing date stamp fields (e.g., MQL Date, SQL Date)<br />• Note any duplicate or conflicting fields tracking similar data<br />• Export field dependencies and workflow rules for reference |
| 3. Establish Baseline Conversion Metrics | 2h | Pull historical data to calculate current conversion rates between stages. End state: Baseline metrics document showing stage-to-stage conversion rates and average time-in-stage.<br /><br />• Pull lead volume by stage for last 90 days from CRM reports<br />• Calculate conversion rates: Lead-to-MQL, MQL-to-SQL, SQL-to-Opportunity, Opportunity-to-Closed Won<br />• Measure average time-in-stage for each lifecycle phase (if date stamps exist)<br />• Identify stages with highest drop-off rates<br />• Compare marketing's reported MQL numbers to sales' accepted SQL numbers<br />• Document any gaps in data (e.g., "no date stamps exist, velocity unmeasurable") |
| 4. Create Current State Assessment Document | 1.5h | Synthesize findings from interviews, CRM audit, and baseline metrics into a single assessment document. End state: Shareable document with gap analysis ready for stakeholder review.<br /><br />• Summarize current lifecycle stage definitions (or lack thereof)<br />• Document misalignments between team definitions<br />• List baseline conversion metrics with data quality caveats<br />• Identify top 3-5 pain points and root causes<br />• Map gaps between current state and best practice lifecycle management<br />• Include recommendations for stage structure in next phase |
| 5. Define Lifecycle Stages and Entry Criteria | 2.5h | Collaboratively define each lifecycle stage with clear, measurable entry criteria that all teams agree to. End state: Documented stage definitions with binary, objective entry criteria for each stage.<br /><br />• Facilitate working session with marketing, sales, and CS stakeholders (60-90 min)<br />• Define standard stages: Subscriber, Lead, MQL, SQL/SAL, Opportunity, Customer, Churned<br />• Establish measurable entry criteria for each stage (e.g., "MQL = Lead Score >= 50 AND visited pricing page")<br />• Define exit criteria and what triggers backward movement (disqualification, recycling)<br />• Document responsible team for each stage (who owns it, who updates it)<br />• Ensure criteria are binary (yes/no) not subjective ("seems ready")<br />• Get verbal sign-off from each stakeholder on definitions |
| 6. Map Handoff Points and SLAs | 1.5h | Define explicit handoff points between teams with response time SLAs. End state: SLA document specifying handoff triggers and expected response times.<br /><br />• Define MQL-to-Sales handoff trigger and SDR/AE response SLA (e.g., 4 hours)<br />• Define SQL acceptance/rejection process and required disposition<br />• Map Opportunity-to-CS handoff for closed-won deals<br />• Document what happens when a lead is recycled (back to nurture)<br />• Establish escalation path for SLA breaches<br />• Define who monitors SLA compliance and how often |
| 7. Design Date Stamp Fields and Velocity Tracking | 1.5h | Design the date stamp architecture to enable time-in-stage and velocity reporting. End state: Field specification document listing all date stamp fields to be created.<br /><br />• List all date stamp fields needed (MQL Date, SQL Date, Opportunity Created Date, Closed Won Date, etc.)<br />• Define what triggers each date stamp (automation vs. manual)<br />• Determine if date stamps should reset on recycling (e.g., re-MQL date)<br />• Plan for historical backfill if possible (use existing dates for current records)<br />• Specify date format and field type in CRM<br />• Document how velocity will be calculated (e.g., days from MQL to SQL) |
| 8. Get Stakeholder Sign-Off on Design | 1.5h | Present the complete lifecycle design to decision-makers for formal approval. End state: Signed-off design document authorizing CRM build.<br /><br />• Prepare lifecycle design presentation (stages, criteria, SLAs, date stamps)<br />• Schedule sign-off meeting with VP Marketing, VP Sales, and RevOps leader<br />• Walk through each stage definition and entry criteria<br />• Confirm SLA agreements are realistic and accepted<br />• Document any requested changes and update design<br />• Get formal approval (email or meeting notes) before proceeding to build |

### Task List: (GTM Lifecycle) 2. CRM Configuration & Reporting
**Contains:** Parts 3-4

| Task | Est | Description |
|------|-----|-------------|
| 9. Create or Update Picklist Fields | 2h | Configure the lifecycle stage picklist field(s) in CRM with agreed-upon values. End state: Single source-of-truth picklist field for lifecycle stages on Lead, Contact, and/or Account objects.<br /><br />• Create or update Lead Status picklist with approved stage values<br />• Create or update Contact Status picklist (if tracking at contact level)<br />• Consider Account-level lifecycle field for ABM motions<br />• Set default values for new records (e.g., "Subscriber" or "New Lead")<br />• Configure field-level security (who can edit, who is read-only)<br />• Add picklist to page layouts and ensure visibility for all users |
| 10. Build Automation Rules for Stage Transitions | 3.5h | Configure automation to move records between stages based on defined entry criteria. End state: Automation rules triggering stage changes when criteria are met.<br /><br />• Build MQL trigger automation (e.g., when Lead Score >= 50 AND pricing page visited, set stage to MQL)<br />• Build SQL handoff automation (e.g., when SDR qualifies lead, set stage to SQL)<br />• Build Opportunity creation automation (e.g., when Opportunity created, update related Lead/Contact to "Opportunity" stage)<br />• Build Closed Won automation (e.g., when Opportunity closed won, set Account to "Customer" stage)<br />• Build recycling automation (e.g., when disqualified, set stage to "Recycled" or "Nurture")<br />• Test each automation in sandbox before deploying to production |
| 11. Configure Date Stamp Fields and Triggers | 3h | Create date fields for each lifecycle stage and automation to populate them. End state: Date stamp fields populated automatically on stage entry.<br /><br />• Create date fields: MQL_Date__c, SQL_Date__c, Opportunity_Date__c, Closed_Won_Date__c, etc.<br />• Build automation to stamp date when stage changes (Process Builder, Flow, or Apex trigger)<br />• Decide handling for re-entry (overwrite date vs. preserve original)<br />• Configure first-touch date stamps where needed (First MQL Date if tracking re-MQLs)<br />• Test date stamps with sample records (advance stage, verify date populates)<br />• Backfill historical records if data exists (use Closed Date for Closed Won, etc.) |
| 12. Build Validation Rules for Stage Integrity | 2h | Create validation rules to prevent invalid stage changes or data quality issues. End state: Validation rules enforcing stage progression logic and data integrity.<br /><br />• Build validation rule to prevent backward movement (e.g., can't go from SQL back to Lead without disposition)<br />• Require disposition reason when disqualifying or recycling leads<br />• Enforce required fields at certain stages (e.g., SQL requires Company Size and Budget Timeline)<br />• Prevent reps from manually overriding automated stage changes (if desired)<br />• Test validation rules to ensure they don't block legitimate use cases<br />• Document validation rules and error messages for user training |
| 13. Create Conversion Rate Reports | 2h | Build reports showing stage-to-stage conversion rates for funnel analysis. End state: Conversion rate reports available to marketing, sales, and RevOps leaders.<br /><br />• Build Lead-to-MQL conversion rate report (MQLs / Total Leads by period)<br />• Build MQL-to-SQL conversion rate report (SQLs / MQLs by period)<br />• Build SQL-to-Opportunity conversion rate report<br />• Build Opportunity-to-Closed Won win rate report<br />• Add filtering by lead source, campaign, rep, and time period<br />• Create report folders with appropriate sharing settings |
| 14. Build Velocity and Time-in-Stage Reports | 2h | Create reports showing average time records spend in each stage. End state: Velocity reports enabling funnel speed optimization.<br /><br />• Build average time from Lead to MQL report (using date stamp fields)<br />• Build average time from MQL to SQL report<br />• Build average time from SQL to Closed Won report<br />• Add segmentation by lead source, deal size, and rep<br />• Identify outliers (leads stuck in stage > X days) for follow-up<br />• Create velocity trend report showing improvement over time |
| 15. Create Lifecycle Dashboard | 2h | Consolidate reports into a single dashboard for executive visibility. End state: Executive-ready dashboard showing funnel health and velocity metrics.<br /><br />• Create dashboard with conversion rate charts by stage<br />• Add velocity metrics showing average time-in-stage<br />• Include volume metrics (leads entering each stage by week/month)<br />• Add leaderboard or breakdown by rep/team if applicable<br />• Configure dashboard refresh schedule (daily or real-time)<br />• Set dashboard visibility for marketing, sales, and CS leaders |
